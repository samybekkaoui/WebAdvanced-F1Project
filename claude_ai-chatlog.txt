Good question. Let me break it down.

The problem: module scope vs global scope

When you use <script type="module">, JavaScript treats the file as a module. Modules have their own private scope — variables and functions defined inside them are not automatically available on the window object (which is the global scope that the browser uses).

Inline onclick attributes in HTML run in the global scope. So when the browser sees:

html
<button onclick="toggleFavDriver('max_verstappen')">
It looks for toggleFavDriver on window. It's not there → silent error → nothing happens.

What window.toggleFavDriver = toggleFavDriver does

It takes your function (which exists privately in the module) and manually attaches it to the global window object. After that line runs, window.toggleFavDriver exists, and the inline onclick can find it.

A cleaner long-term alternative would be to avoid onclick in HTML strings altogether and instead use event delegation — attach one listener to the parent container and check what was clicked:

js
driversList.addEventListener('click', (e) => {
  const btn = e.target.closest('.star-btn');
  if (!btn) return;
  toggleFavDriver(btn.dataset.id);
});
And in your HTML string, put the id in a data- attribute instead of an onclick:

js
`<button class="star-btn" data-id="${driver.driverId}">★</button>`
This is the more "correct" approach for modules because you never need inline handlers — the listener stays in module scope where your functions live. But the window. fix is perfectly valid and simpler for where you are now.

Ja, precies! CSS heeft twee mechanismes die bepalen welke regel "wint":

1. Specificity (gewicht van de selector) Een meer specifieke selector wint altijd, ongeacht volgorde. Bijvoorbeeld #id verslaat altijd .class, en .class verslaat altijd element.

2. Cascade (volgorde) — bij gelijk gewicht Als twee regels even specifiek zijn, wint degene die later in het bestand staat. Dat was jouw probleem:

css
/* Beide even specifiek: gewoon ".hamburger" */

@media (max-width: 540px) {
  .hamburger { display: flex; }  /* positie 1 */
}

.hamburger { display: none; }   /* positie 2 — wint altijd! */
De media query verandert niet de specificity — .hamburger binnen een media query is even zwaar als .hamburger buiten. Dus de latere display: none won altijd, op élke schermgrootte.

Vuistregel voor responsive CSS:

Schrijf altijd je basis/desktop stijlen eerst, en zet media queries onderaan. Dan kunnen de media queries de basis-stijlen overschrijven zonder verrassingen:

css
/* 1. Basis stijlen */
.hamburger { display: none; }

/* 2. Tablet overrides */
@media (max-width: 860px) { ... }

/* 3. Mobile overrides — altijd als laatste */
@media (max-width: 540px) {
  .hamburger { display: flex; } /* wint nu correct */
}
Dit is ook de "mobile-first" vs "desktop-first" aanpak — veel developers schrijven zelfs hun basis stijlen voor de kleinste schermen en gebruiken dan min-width media queries om groter te maken. Maar bij desktop-first zoals jij doet, is de volgorde bovenaan basis → onderaan overrides de gouden regel.

Good question! Here's the flow:

The Jolpica API returns each driver/team with a url field that looks like this:

https://en.wikipedia.org/wiki/Max_Verstappen
The fetchWikiSummary function does two things with that string:

js
const title = decodeURIComponent(wikiUrl.split('/wiki/')[1]);
Step 1 — .split('/wiki/')[1] Splits the string on the literal text /wiki/, which gives you an array of two parts:

["https://en.wikipedia.org", "Max_Verstappen"]
Taking index [1] gives you just "Max_Verstappen".

Step 2 — decodeURIComponent(...) Some names have special characters in the URL, like "Jos%C3%A9_Mar%C3%ADa_L%C3%B3pez". decodeURIComponent converts those escape sequences back to "José_María_López" so Wikipedia recognises it.

Then that title gets dropped straight into the Wikipedia REST API URL:

js
const apiUrl = `https://en.wikipedia.org/api/rest_v1/page/summary/${encodeURIComponent(title)}`;
encodeURIComponent re-encodes it cleanly for the API request (spaces → %20, etc.). Wikipedia then returns a JSON object with extract (the bio text) and thumbnail.source (the image URL) — and those just get dropped into the modal elements.

So the whole trick is that Jolpica conveniently stores the exact Wikipedia URL already, meaning you don't have to search for the right page yourself — you just extract the title from what you already have.
